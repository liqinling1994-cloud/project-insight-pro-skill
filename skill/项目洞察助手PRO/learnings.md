# 项目洞察助手PRO 经验沉淀

## 用户反馈

- 2026-06-23：PRO 版只保留项目洞察第一阶段，产出完整“项目宣传写作输入包”，不承接选题框架、素材筛选和写作成稿。
- 2026-06-23：旧版输出编号很多但表达分散。用户需要先看到“产品是什么、分析结论、适合对象、目标群体、日常推广应该打哪些人”，再看编号、证据和红线。
- 2026-06-23：默认输出仍有大量编号和置信度会影响正常阅读。以后默认输出必须去掉编号、置信度和审计附录；只有用户明确要求审计版、证据版、编号版、置信度表时才展开。
- 2026-06-23：项目拆解不能只停在产品、价值、人群、推广四项；默认还要补交易与交付结构、结果路径与边界、用户决策链、成交阻力、内容资产和销售转化入口。
- 2026-06-23：章节标题也要减少机械数字感。默认报告优先使用自然标题，事实编号只作为内部底稿或按需审计附录。

## 最佳实践

- 第一阶段内部底稿必须保留来源链、风险边界和必要审计信息；默认可见报告不展示编号和置信度。
- 正式输出采用“默认报告 + 按需审计附录”：默认报告先给产品判断、人群、交易交付、结果边界、决策链、成交阻力和推广打法；审计附录只在用户明确要求时承载编号、来源、置信度和信息索引。
- 默认交付使用“干净阅读版”：自然语言结论 + 少量业务表格 + 来源短注 + 补证清单。编号和置信度作为内部底稿，不进入默认可见报告。
